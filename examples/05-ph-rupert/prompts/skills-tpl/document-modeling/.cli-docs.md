## Document Modeling

Design and implement Powerhouse document models — the schema layer that defines state, operations, and reducer logic for a document type in the Reactor ecosystem. All document model code is generated through the Vetra MCP; this skill guides the agent through the full creation workflow from schema design to reducer implementation.

## When to use

Use this skill when you need to create a new document model from scratch, add operations or modules to an existing model, redesign a state schema, or implement reducer business logic. Typical prompts: "Design an invoice document model with line items and tax calculations" or "Add a SET_DUE_DATE operation to the invoice model."

## Parameters

- **prompt** (required): Description of the document model to create or modify.
- **mode** (default: `expert`):
  - `expert` — The agent asks pointed technical questions to confirm major design decisions (state schema shape, module boundaries, operation granularity) and presents a proposal for approval before proceeding. Best for developers familiar with Powerhouse concepts.
  - `discovery` — The agent explains document modeling concepts as it goes, asks design questions in plain language, and guides you through decisions step by step. Best for users new to Powerhouse or document-driven architectures.
  - `one-shot` — The agent makes all design decisions autonomously based on the prompt and executes the full workflow without asking. Best when you have a clear specification and want fast results.

## Example invocations

```
/document-modeling --prompt "Design an invoice document model with line items, tax rates, and payment status"
/document-modeling --mode discovery --prompt "I need a way to track project tasks with assignments and deadlines"
/document-modeling --mode one-shot --prompt "Create a simple contact list model with name, email, phone, and company fields"
```

## Workflow

The agent follows a structured multi-phase process:

### Phase 1: Check prerequisites
Verifies the Reactor Package project is running via `{{commands.reactor-project-ps.id}}`, reviews any existing specifications or stakeholder requirements, and confirms the `{{services.reactor-project.mcpPrefix}}*` MCP tools are available and responsive.

### Phase 2: Write description
Creates the document model's description, including document type name, header fields, and author metadata using `{{services.reactor-project.mcpPrefix}}createDocument` and `{{services.reactor-project.mcpPrefix}}addActions`. This establishes the identity of the new model in the Vetra system.

### Phase 3: Define state and operations
Designs the global state schema using GraphQL type definitions (with Powerhouse scalar types like OID, PHID, Amount, DateTime). Organizes operations into logical modules, defines input types for each operation, and adds error definitions. All definitions are submitted through `{{services.reactor-project.mcpPrefix}}addActions` with actions like `SET_STATE_SCHEMA`, `ADD_MODULE`, `ADD_OPERATION`, and `ADD_OPERATION_ERROR`. Checks `{{commands.reactor-project-logs.id}}` to confirm the code generator ran.

### Phase 4: Implement reducers
After the code generator produces boilerplate files, the agent implements business logic in `src/reducers/*.ts`. Reducers must be pure synchronous functions — no `Date.now()`, `crypto.randomUUID()`, or side effects. All dynamic values come from `action.input`. The agent also submits reducer code via `SET_OPERATION_REDUCER` through `{{services.reactor-project.mcpPrefix}}addActions` so future code generations stay in sync.

### Phase 5: Quality assurance
Runs `npm run tsc` and `npm run lint:fix` to validate the implementation, then communicates results to stakeholders.
