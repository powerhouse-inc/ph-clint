# Visual QA - Skill Preamble

With this skill, you perform visual quality assurance on the HTML outputs (site and/or deck). You screenshot, identify issues, fix them, and verify the fixes.

## Interaction Mode

**expert mode**: Present issues found and proposed fixes for review before applying. Discuss trade-offs for complex fixes.

**discovery mode**: Explain what visual QA checks for and why each check matters. Walk through issues and fixes.

**one-shot mode**: Execute the full QA cycle autonomously: screenshot → identify → fix → verify. Report results.

## Critical Guards

### 1. Wait for Animations
Use the `animationWait` config value (default 2000ms) to pause after page load before capturing. CSS transitions, fade-ins, and layout shifts need time to complete. A screenshot of a half-animated page is useless.

### 2. Check Multiple Widths
Screenshot at minimum three viewport widths:
- **Mobile**: 375px (iPhone SE)
- **Tablet**: 768px (iPad)
- **Desktop**: 1440px (standard laptop)

Some issues only appear at specific widths.

### 3. Contrast Ratios
Don't rely on visual inspection alone. For key text elements, verify:
- Body text: minimum 4.5:1 ratio against background
- Large text (>18px or >14px bold): minimum 3:1 ratio
- CTA buttons: text meets contrast against button background

### 4. Text Orphans
Single words on their own line at the end of paragraphs or headlines look unprofessional. Fix with CSS `white-space: nowrap` on the last two words, or adjust copy length.

### 5. Multiple Passes
Fixes can introduce new issues (especially responsive fixes). After each fix round, re-screenshot and re-check. Plan for 2-3 passes.

### 6. Document Everything
Create an issue log: location (page + section), viewport width, description, screenshot filename, fix applied, verification status.

## QA Outputs

All outputs go to `screenshots/`:
- `{page}-{width}px.png` — Screenshots at each viewport
- `{page}-{width}px-pass{N}.png` — Verification screenshots
- `issues.md` — Issue log with resolution status
