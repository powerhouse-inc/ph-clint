# ph-clint leaks `any` across the integration surface

> **Implementation plan:** see [`specs/stricter-typing/README.md`](../stricter-typing/README.md) for the phased fix (framework types → binding helper → document-change trigger helper → codegen registry).

## Problem

The ph-clint framework is designed to drive CLIs that glue together Powerhouse document models, the Reactor client, triggers, services, and a Mastra agent. Implementation projects are expected to extend all of these, typically with strongly-typed, codegen-produced document models (e.g. `ph-clint-app/document-models/ph-clint-project/v1`). In practice, whenever an impl project crosses the ph-clint boundary — especially into the Powerhouse integration — **the type information is erased**. The impl has to either:

1. Re-declare a minimal shape of the document state by hand, or
2. Cast `any` / `unknown` results and lose autocomplete, refactor safety, and the runtime guarantees of the generated `isXxxDocument` guards.

This is the opposite of the expected outcome. Working with a typed document state inside a trigger, service, or command should be the path of least resistance — ideally the *only* path, short of deliberately opting out.

## Current behavior (evidence from this repo)

The `spec-change` trigger in `packages/ph-clint-cli/ph-clint-cli/src/triggers/spec-change.ts` — written today against the current ph-clint API — ended up with these casts:

```ts
// 1. reactor.client is typed `any`.
//    No IDE help for .get, .getChildren, .subscribe, or any other method.
const raw = await reactor.client.get(docId);

// 2. The returned document is `any`. We have to import the generated guard
//    ourselves, from a long impl-package path, and narrow manually.
if (!isPhClintProjectDocument(raw)) return null;
const doc: PhClintProjectDocument = raw;

// 3. Config on the TriggerContext defaults to `Record<string, unknown>`.
//    The projectDocumentId field we added to configSchema is invisible here.
const cfg = ctx.context.config as { projectDocumentId?: string } | undefined;

// 4. Trigger state is `Record<string, unknown>`. Every read/write needs a cast.
ctx.state.pending = 0 as number;
// …
const pending = ctx.state.pending as number;

// 5. Event payloads are `unknown`. We reverse-engineer the shape by logging
//    the raw JSON and guessing keys.
on('powerhouse:document:changed', (payload) => {
  const docs =
    (payload as { documents?: Array<{ id?: unknown; header?: { id?: unknown } }> } | undefined)
      ?.documents ?? [];
  // …
});
```

Inside the framework itself, the same erasure is explicit:

```ts
// packages/ph-clint/src/integrations/powerhouse/types.ts
export interface ReactorContext {
  /** The Reactor client — full CRUD + subscription API (IReactorClient). */
  client: any;
  // …
  _module?: any;
}

export interface PowerhouseIntegrationOptions {
  documentModels: any[];
  // …
}

// packages/ph-clint/src/integrations/powerhouse/subscriptions.ts
export function bridgeSubscriptions(
  client: any,
  subscriptions: SubscriptionConfig,
  emit: (event: string, data?: unknown) => void,
): () => void {
  return client.subscribe(search, (event: any) => { /* … */ });
}

// packages/ph-clint/src/core/types.ts
export interface CommandContext<TConfig = Record<string, unknown>> {
  config: TConfig;                        // generic exists…
  emit?: (event: string, data?: unknown) => void; // …but events are stringly-typed
  on?: (event: string, handler: (data?: unknown) => void) => void;
  reactor?: () => Promise<ReactorContext | undefined>; // …and reactor is any
}

export interface TriggerContext {          // no TConfig, no TState, no TReactor
  context: CoreContext;                    // ⇒ config is Record<string, unknown>
  state: Record<string, unknown>;          // ⇒ state reads require casts
  reactor: () => Promise<ReactorContext | undefined>;
}
```

The `any`s on `client`, `_module`, and `documentModels` are deliberate — the comment at the top of `types.ts` notes that `@powerhousedao/reactor` is an optional peer dependency and "we can't import its types at the module level". That justifies runtime decoupling but not type erasure: a `type-only` import (or a generic parameter) would satisfy both constraints.

## Impact

Every impl project — today `ph-clint-cli`, tomorrow everything generated by it — pays the same "untype tax":

- **No IDE support** across the ph-clint boundary. `reactor.client.` shows no members; `ctx.context.config.` shows nothing from the impl's `configSchema`.
- **Silent breakage on upstream changes.** When the reactor client API shifts (as `event.documents[].id` vs `.header.id` already did in this session), impls find out at runtime by logging payload JSON, not at compile time.
- **Handwritten shapes drift.** `PhClintProjectDocumentState` in `src/spec/from-document.ts` was a minimal hand-typed mirror of the generated `PhClintProjectGlobalState`, written because importing the real type was non-obvious. Those mirrors get out of sync the moment a codegen regenerates.
- **Codegen can't help.** ph-clint-cli scaffolds triggers and services for the impl — but since the framework types are `any`, the scaffolded code can only emit `any` too, or hardcode `as`-casts that break on regeneration.

## Root cause

ph-clint's type surface is **uniformly untyped at the integration boundary**, by a mixture of:

1. **`any` by convenience.** `ReactorContext.client: any`, `documentModels: any[]`, and the subscription bridge's `client: any, event: any` are the clearest examples. They sidestep "optional peer dep" concerns at the cost of never recovering types in consumers.
2. **Generics that don't propagate.** `CommandContext<TConfig>` exists but `TriggerContext` drops it; `CoreContext<TConfig>` is a type alias that nothing carries through to triggers, services, or preflight checks downstream of the core pipeline.
3. **Stringly-typed event bus.** `emit(event: string, data?: unknown)` and the matching `on()` accept any event name and give back `unknown`. Well-known, framework-emitted events like `powerhouse:document:changed`, `powerhouse:document:created`, and `powerhouse:document:deleted` have no declared payload shape.
4. **Untyped state container on `TriggerContext`.** Every trigger builds up its own per-instance state (`pending` counters, queues, timers) but the framework gives it a `Record<string, unknown>` to live in.
5. **No canonical "registered document models" type.** The impl passes `documentModels: [PhClintProjectV1, …]` into `buildDefaultReactor` but the framework discards that information. There is no `DocumentRegistry<T>` type that maps `documentType` strings to document shapes, so `reactor.client.get<T>(…)` can't be inferred from the registry either.

## Expected outcome

Working with a typed document state from an impl project should be **the default path**. Specifically:

1. `ctx.context.config` inside any CLI-authored callback (command, trigger, service, preflight, agent setup) is typed as the impl's `z.infer<typeof configSchema> & z.infer<typeof secretsSchema>`.
2. `ctx.reactor()` returns a `ReactorContext<Registry>` where `Registry` is derived from the `documentModels` passed at `configureReactor` time. `reactor.client.get(id)` returns a discriminated union of registered document types (or `Registry[TType]` when called as `get<TType>(id)`).
3. The event bus has a known payload map for framework-emitted events: `on('powerhouse:document:changed', handler)` gives `handler` a typed `DocumentChangeEvent` payload with concrete `documents: Array<{ id: string; header: { id: string; documentType: RegisteredType } }>` based on the registry.
4. `defineTrigger<TState, TConfig, TDocument>()` / `defineService<TConfig>()` are fully generic. `ctx.state` is typed as `TState`; `ctx.context.config` is `TConfig`; no casts.
5. A first-class helper like `createDocumentChangeTrigger<TDoc>({ documentType, documentId?, onChange })` exists in `@ph-clint/powerhouse` so the common pattern (event filter + doc load + typed state callback) is a one-liner. The callback receives `TDoc['state']['global']` directly.
6. Impl projects **do not import from deep ph-clint-app paths**. Generated reactor packages should re-export `DocumentRegistry`, action creators, and type guards from a top-level `@impl/types` entry; ph-clint-cli's codegen consumes that registry and emits triggers/services against it.
7. The `any` in `ReactorContext.client` is gone. If avoiding the runtime dep of `@powerhousedao/reactor` matters, `ReactorContext` becomes `ReactorContext<TClient = IReactorClient>` with `TClient` imported `import type` (erased at runtime, present in declarations).

## Suggested fix

Land this in three layered passes, each testable on its own:

### 1. Strict types at the framework boundary (no API break)

- Replace `any` with `import type`:
  - `ReactorContext.client: any` → `import type { IReactorClient } from '@powerhousedao/reactor-browser';` and use that type (or a generic `TClient = IReactorClient`).
  - `documentModels: any[]` → `DocumentModelModule[]` (already imported via `document-model` in user code).
  - `subscriptions.ts` — remove `client: any, event: any` in favor of the real `IReactorClient` + `SubscriptionEvent` types.
- Add a `PhClintEvents` interface for framework-emitted events. Type `emit`/`on` as overloads keyed on that interface plus a fallback for arbitrary user events:
  ```ts
  interface PhClintEvents {
    'powerhouse:document:changed':  DocumentChangeEvent;
    'powerhouse:document:created':  DocumentCreatedEvent;
    'powerhouse:document:deleted':  DocumentDeletedEvent;
  }
  function on<K extends keyof PhClintEvents>(event: K, handler: (data: PhClintEvents[K]) => void): void;
  function on(event: string, handler: (data?: unknown) => void): void;
  ```

### 2. Propagate generics through the trigger / service / command pipeline

- Lift `TConfig` onto `TriggerContext<TState, TConfig>` and `ServiceContext<TConfig>`. Today `CommandContext<TConfig>` already exists; the trigger/service contexts should be the same generic shape.
- Add `defineTrigger<TState, TConfig = …, TDoc = …>({ state, … })` with a `state: () => TState` initializer. `ctx.state` is then `TState`, no casts.
- Let `defineCli({ configSchema })` auto-infer `TConfig` and propagate it into every callback the CLI registers. Today this works for commands but not triggers.

### 3. A typed Powerhouse helper layer

- Introduce a `DocumentRegistry<T>` type: `Record<string /* documentType */, { state: Record<string, unknown>; actions: unknown; document: PHDocument<…> }>` that impl projects produce (or have codegen produce) from their doc-model modules.
- `configureReactor<Registry>({ create, … })` captures the registry type and threads it into `ReactorContext<Registry>`:
  ```ts
  interface ReactorContext<R extends DocumentRegistry = DocumentRegistry> {
    client: TypedReactorClient<R>;
    driveId: string;
    // …
  }
  interface TypedReactorClient<R> {
    get<T extends keyof R>(id: string, type?: T): Promise<R[T]['document'] | null>;
    subscribe<T extends keyof R>(
      search: { documentTypes?: T[] },
      cb: (event: { type: string; documents: Array<R[T]['document']> }) => void
    ): () => void;
    // …
  }
  ```
- Ship `createDocumentChangeTrigger<T extends keyof R>({ documentType, documentId?, onChange })` in `ph-clint/integrations/powerhouse`. The callback is handed `R[T]['document']['state']['global']` — no imports of `PhClintProjectGlobalState`, no guards, no casts. This one helper collapses `specChangeTrigger` + example 06's `createDocumentChangeTrigger` into two-line trigger definitions.
- ph-clint-cli's codegen then emits:
  - A `document-registry.ts` in the impl that unions the registered document models into `Registry`.
  - Triggers/services scaffolded against `createDocumentChangeTrigger<T>` rather than hand-written event handlers.

## Workaround (today)

- Manually cast `ctx.context.config as MyConfig`, `ctx.state.x as number`, and payloads on `on('powerhouse:document:changed', …)`.
- Import generated types from the impl package using long paths (`ph-clint-app/document-models/<name>/v1/gen/types`) and call `isXxxDocument()` guards on `reactor.client.get()` results.
- Write a minimal mirror type (`PhClintProjectDocumentState`) co-located with the trigger to hide the cast in one place.

None of these scale. Each one is a place where the impl has to know something the framework could have told it.

## Scope / priority

This is cross-cutting but low-risk:

- Steps 1 and 2 are additive. Existing impls keep working because the generics all default to today's loose types.
- Step 3 lands as a new module (`ph-clint/integrations/powerhouse/typed`) and can be adopted per-trigger.
- The payoff compounds: every generated impl project (ph-clint-cli phase 5 onwards, and anything downstream like ph-mara) inherits the typed surface for free.
