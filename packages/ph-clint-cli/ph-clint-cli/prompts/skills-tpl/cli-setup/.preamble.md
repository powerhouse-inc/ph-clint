# CLI Setup - Skill Preamble

With this skill, you can scaffold or extend a `defineCli()` configuration — the composition root that wires commands, config, services, triggers, and integrations into a single CLI runtime. A single `defineCli()` call produces a CLI that works in command mode (`mycli cmd --arg val`), interactive REPL mode (`/cmd --arg val`), and as an MCP server — all from the same code.

## Core Principles

### defineCli Is Declarative

You describe _what_ the CLI has — the framework wires it up. The order of options doesn't matter, but understanding what triggers what does:

- `configSchema` present → auto-injects a `config` command for viewing/modifying settings
- `secretsSchema` present → fields merge into configSchema internally, but are censored in output
- `services` present → auto-injects per-service management commands (`{id}-start`, `{id}-stop`, etc.)
- `interactive` present → enables the `-i` flag for REPL mode
- `triggers` present → enables routine loop
- `cli-docs` command is always auto-injected (CLI help for the agent)

### Resolvable Values

`description` and `interactive.welcome` accept either a static string or a function `({ config, workdir }) => string`. Use functions when the message depends on runtime state — e.g., showing "demo mode" vs "LLM mode" based on whether an API key is configured.

### The Cli Object

`defineCli()` returns a `Cli` object — the public API with methods for running, testing, introspection, and metadata. The standard entry point is `cli.run(process.argv)`. For programmatic access (e.g., codegen scripts), use `cli.bootstrap()` which resolves workdir, config, and context without entering the dispatch loop.

### Two Interaction Modes

1. **Command mode** (headless): Execute one subcommand, then exit. Triggered by `mycli cmd --flag val`.
2. **Interactive REPL mode**: Persistent session with `/command` syntax and bare text for the default subcommand (typically the agent). Triggered by `mycli -i` or when keep-alive features are active (routine loop, API server).

### 6-Layer Config Resolution

Config values are resolved in this priority order (highest wins):
1. `--config <file>` flag
2. Environment variables: `{CLI_NAME}_{UPPER_SNAKE_FIELD}`
3. Local config: `{workdir}/.ph/{cliName}.config.local.json`
4. User config: `~/.ph/{cliName}.config.user.json`
5. Implementation defaults: `configDefaults` in CliOptions
6. Zod schema `.default()` values

### Config vs Secrets

- `configSchema` — user-facing settings (model name, ports, feature flags). Shown in `config` command output.
- `secretsSchema` — sensitive values (API keys, tokens). Merged into configSchema internally but censored as `***` in output unless `--revealSecrets` is used.

Put API keys and tokens in `secretsSchema`, not `configSchema`.

### configDefaults vs Schema Defaults

- `.default(value)` in the Zod schema → layer 6, the fallback when nothing else is configured
- `configDefaults` in CliOptions → layer 5, implementation-level overrides that sit above schema defaults but below user/local config

Use schema `.default()` for universal sensible defaults. Use `configDefaults` for implementation-specific overrides that differ from the general-purpose default.

### Entry Point Convention

The CLI entry point is a TypeScript file with a Node.js shebang:

```typescript
#!/usr/bin/env node
import { defineCli } from '@powerhousedao/ph-clint';
// ... imports ...
const cli = defineCli({ /* options */ });
cli.run(process.argv);
```

The `package.json` needs a `"bin"` field pointing to the compiled output and `"type": "module"` for ESM.

## Common Pitfalls

- **Forgetting `version`** — Commander needs it for the `--version` flag; omitting it breaks help output
- **Putting secrets in `configSchema`** — they show up in `config` command output and metadata, exposing sensitive values
- **Not setting `interactive.welcome`** — REPL starts with no context, confusing the user
- **Using `configDefaults` for universal defaults** — use `.default()` in the schema instead; `configDefaults` is for implementation-level overrides only
- **Calling `configureAgent()` inside `defineCli()`** — it's a method on the returned `Cli` object, called after definition
- **Forgetting `"type": "module"` in package.json** — ESM imports fail at runtime
- **Not exporting the `cli` object** — other files (e.g., `mastra/index.ts`) may need it for `cli.bootstrap()`

## When to Use `createTypes()`

For projects with a config schema, use the typed factory to get automatic config typing in all commands:

```typescript
import { createTypes } from '@powerhousedao/ph-clint';
const { defineCommand, defineTrigger } = createTypes({ configSchema });
// context.config is now fully typed in every command
```

## Codegen-Managed Projects

In ph-clint-cli projects, `{{commands.clint-project-init.id}}` generates the initial scaffold including `cli.ts` with `@clint:begin`/`@clint:end` marker regions. The codegen system:

- **Generates `cli.ts`** with marker regions for imports, commands, services, triggers, prompts, events, interactive, reactor, and mastra
- **Generates `framework.gen.ts`** (codegen-owned) with the document registry — regenerated when the spec's `documentTypes` change
- **Emits `framework.ts`** once (user-owned) with `configSchema`, `secretsSchema`, and `createTypes()` — never overwritten after creation

**The agent never hand-edits marker regions in `cli.ts`.** To add or change CLI features:

1. Update `project-spec.json` to declare the new structure
2. Run `{{commands.clint-project-regen.id}}` to regenerate the marker regions
3. Run `{{commands.clint-project-build.id}}` to verify

User-owned code outside markers (`configSchema`, `secretsSchema`, `CLI_NAME`) is preserved across regens. User-owned files (`framework.ts`, `src/commands/*.ts`, `src/services/*.ts`, `src/triggers/*.ts`) are never overwritten.
