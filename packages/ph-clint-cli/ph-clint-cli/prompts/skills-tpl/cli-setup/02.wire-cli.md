# Phase 2: Wire the defineCli Call

Compose the `defineCli()` call with all the options determined in Phase 0.

## Steps

### 1. Assemble the core options

```typescript
import { defineCli } from '@powerhousedao/ph-clint';
import { configSchema, secretsSchema } from './config.js';
import { myCommand, otherCommand } from './commands/index.js';

export const cli = defineCli({
  name: 'my-app',
  version: '1.0.0',
  description: 'My application CLI',
  commands: [myCommand, otherCommand],
  configSchema,
  secretsSchema,
});
```

Always export the `cli` object — other files may need it (e.g., `mastra/index.ts` for `cli.bootstrap()`).

### 2. Configure interactive mode

Add an `interactive` block if the CLI supports REPL mode:

```typescript
interactive: {
  welcome: 'My App CLI — type /help for commands',
},
```

For dynamic welcome messages that depend on config:

```typescript
interactive: {
  welcome: ({ config, workdir }) => {
    const mode = config.apiKey ? config.model : 'demo mode';
    return `My App (${mode})\nWorkdir: ${workdir}`;
  },
},
```

### 3. Add services (if needed)

```typescript
import { myService } from './services/my-service.js';

defineCli({
  // ...
  services: [myService],
});
```

This auto-injects `my-service-start`, `my-service-stop`, `my-service-restart`, `my-service-ps`, `my-service-logs`, and `my-service-manage` commands.

### 4. Add event handlers (if needed)

Wire lifecycle events for services or custom events:

```typescript
events: {
  'service:ready': (event, log) => {
    log.info(`${event.name} is ready`);
  },
  'service:failed': (event, log) => {
    log.error(`${event.name} failed: ${event.error}`);
  },
  'service:stopped': (event, log) => {
    log.info(`${event.name} stopped`);
  },
},
```

Available service events: `service:pattern-matched`, `service:ready`, `service:failed`, `service:restarting`, `service:stopped`. Each receives `(event, log)` where `log` is a structured Logger.

### 5. Add prompts config (if skills are needed)

```typescript
import path from 'node:path';

prompts: {
  artifacts: [path.join(CLI_ROOT, 'gen', 'skills')],
  agents: {
    'my-agent': {
      name: 'MyAgent',
      sections: ['AgentBase.md', 'MySpecialization.md'],
      skills: ['skill-one', 'skill-two'],
    },
  },
  skills: {
    'skill-one': {
      description: 'Description of skill one',
      inputSchema: z.object({
        mode: z.enum(['expert', 'discovery']).default('expert').describe('Interaction mode'),
      }),
      instructionTemplate: 'Use your \{{skillId}} skill in \{{mode}} mode for: \{{prompt}}',
    },
  },
},
```

### 6. Call configureAgent / configureReactor (if needed)

These are called AFTER `defineCli()` — they are methods on the returned `Cli` object:

```typescript
export const cli = defineCli({ /* ... */ });

// Agent integration (Mastra)
cli.configureAgent(createAgent);

// Reactor integration (Powerhouse)
cli.configureReactor({
  registry: [/* document models */],
  // ...
});
```

Both are lazy-loaded — the actual agent/reactor is only created when first accessed.

### 7. Codegen projects: use spec + regen

In codegen-managed projects, do NOT manually edit `cli.ts` marker regions. Instead:

1. Update `project-spec.json` with the new features (commands, services, triggers, integrations)
2. Run `{{commands.clint-project-regen.id}}` to regenerate marker regions
3. Edit user-owned files directly: `framework.ts` for config/secrets schemas, `src/commands/*.ts` for commands, etc.

Core options outside markers (`name`, `version`, `description`, `configSchema`, `secretsSchema`) are in the user-editable zone — edit those directly in `cli.ts`.

### 8. Verify built-in commands

After wiring, confirm these commands will be auto-injected:
- `cli-docs` — always present
- `config` — present if `configSchema` was provided
- Service commands — present for each entry in `services`
- Skill commands — present for each entry in `prompts.skills`
