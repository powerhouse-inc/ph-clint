# Command Definition - Skill Preamble

With this skill, you can define ph-clint commands — the atomic building blocks of every CLI. A single command definition produces a CLI subcommand (`mycli greet --name World`), a REPL command (`/greet --name World`), an MCP tool, and an agent tool — all from the same code.

## Core Principles

### One Command = One User Intent

Each command does one thing. Don't create CRUD bundles — instead define `add-task`, `list-tasks`, `remove-task` as separate commands. This keeps schemas focused and help text clear.

### inputSchema Is the Single Source of Truth

The Zod `inputSchema` drives everything: CLI flags, help text, validation, agent tool descriptions, and interactive prompting. Design the schema carefully — it's the command's public API.

### `.describe()` on Every Field

Every Zod field MUST have `.describe('...')`. This text appears in:
- CLI `--help` output
- REPL auto-completion hints
- Agent tool descriptions
- Parameter prompting labels

Without `.describe()`, the user sees empty descriptions everywhere.

### Return `{ text, data }` for Agent-Compatible Commands

- Return a **string** for simple display-only commands
- Return `{ text: 'human-readable', data: structuredResult }` when the command will also be used as an agent tool — `text` is shown to users, `data` is consumed by the agent

### Execute Receives Validated Input and Injected Context

The `execute(input, context)` function receives:
1. `input` — already validated and parsed by Zod. Destructure directly: `({ name, loud })`
2. `context` — injected by the framework. Destructure only what you need: `{ workspace, config }`

Never construct context yourself. Never call `process.exit()` or `console.log()` — use the injected `context.stdout()` for streaming output, and throw errors for failures.

## Common Pitfalls

- **Forgetting `.describe()`** — help text and tool descriptions show empty fields
- **Using `.optional()` when you mean `.default('value')`** — optional means the flag can be omitted entirely with no fallback; default provides a sensible fallback value
- **Returning void** — the CLI has nothing to display; always return a string or `{ text, data }`
- **Accessing `context.services` without null check** — it's `undefined` when no services are configured
- **Using `console.log`** — use `context.stdout(text)` instead so output is capturable in tests
- **Calling `process.exit()`** — throw an error instead; the framework handles exit codes

## When to Use Typed Generics

For simple commands, `defineCommand({ ... })` with inline schemas infers types automatically. Use explicit generics when you need to:
- Share the input schema across commands (extract to a const)
- Enforce a specific output type
- Bind to a project-specific config type via `createTypes()`

```typescript
// Project-level typed factory (from framework.ts)
const { defineCommand } = createTypes({ configSchema });

// Now context.config is typed automatically
const myCmd = defineCommand({
  id: 'my-cmd',
  inputSchema: z.object({ name: z.string().describe('Name') }),
  execute: async ({ name }, { config }) => {
    // config is fully typed here
  },
});
```

## Working in a Codegen-Managed Project

ph-clint-cli projects are spec-driven. When defining commands:

- **Import `defineCommand` from `../framework.js`**, not from `@powerhousedao/ph-clint`. The typed factory from `framework.ts` gives automatic config typing via `createTypes()`.
- **Never hand-edit `cli.ts` marker regions.** The `@clint:begin`/`@clint:end` regions are codegen-owned. Instead:
  1. Write the command file in `src/commands/`
  2. Update `project-spec.json` to declare the new command
  3. Run `{{commands.clint-project-regen.id}}` to regenerate `cli.ts` with the correct imports and registration
- **Build** after regen with `{{commands.clint-project-build.id}}` to verify TypeScript compiles
