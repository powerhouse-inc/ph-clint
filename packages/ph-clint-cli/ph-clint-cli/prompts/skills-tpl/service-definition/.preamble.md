# Service Definition - Skill Preamble

With this skill, you can define ph-clint services — long-running background processes with lifecycle management, readiness detection, preflight validation, and automatic restart. A single `defineService()` call produces auto-generated management commands (`{id}-start`, `{id}-stop`, `{id}-restart`, `{id}-ps`, `{id}-logs`, `{id}-manage`) and integrates with the event bus for cross-cutting concerns.

## Core Principles

### Services vs Processes

Services are long-running with lifecycle management (start, stop, restart, health checks). One-shot shell commands use `context.processes.run()` inside a command's execute function instead. If it runs and exits, it's a process. If it stays alive and serves requests, it's a service.

### Readiness Is a Contract

The service declares what "ready" looks like via stdout regex patterns. The framework watches stdout AND stderr combined, matching each line against the patterns. ALL patterns must match before the service transitions to `ready`. If any pattern never matches, the service hits the readiness timeout and is marked `failed`.

### Endpoint Captures Extract URLs from Regex Groups

Captures turn regex groups into named endpoints. Two forms:

- **Simple**: `{ 'name': groupIndex }` — captures the regex group as a plain string
- **Typed**: `{ 'name': { group: index, type: 'api-mcp' } }` — adds an `EndpointType` that enables downstream features like MCP tool discovery

Capture groups are 1-indexed (group 0 is the full match). The `type` field accepts: `'other'`, `'api-mcp'`, `'api-rest'`, `'api-graphql'`, `'website'`.

### Preflight Checks Fail Fast

Preflight checks run before the process spawns. They validate preconditions (working directory, required binaries, port availability) and fail with clear messages rather than letting a process spawn and fail cryptically. Order checks from cheapest to most expensive: workdir first, then binaries, then ports (network I/O).

### Shutdown and Restart Policies

- **Shutdown**: `{ signal, timeout }` — sends `signal` (SIGTERM, SIGINT, or SIGKILL) then waits `timeout` ms. After timeout, SIGKILL is sent unconditionally.
- **Restart**: `{ enabled, maxRetries, delay }` — on crash, automatically restart up to `maxRetries` times with `delay` ms between attempts.

### maxInstances Controls Concurrency

Use `maxInstances: 1` (default) for exclusive services like dev servers. Use higher values for services that can run multiple instances in parallel (e.g., worker pools).

## Service State Machine

Services transition through these states:

```
idle → starting → ready → stopping → stopped
                ↘ failed → (restarting → starting)
```

- **idle**: Defined but never started
- **starting**: Process spawned, waiting for readiness patterns to match
- **ready**: All readiness patterns matched (or no readiness config), service is accepting connections
- **failed**: Readiness timeout expired, or process crashed during startup
- **stopping**: Shutdown signal sent, waiting for process to exit
- **stopped**: Process has exited (clean shutdown or after force-kill)
- **restarting**: Failed service with restart policy, waiting for delay before retry

Each transition emits an event via the event bus (if configured).

## Instance Identity

Every running service has an `instanceId` derived from the start parameters:

- No name, no workdir → instanceId = `serviceId`
- Named instance → instanceId = `serviceId:name`
- Workdir-derived → instanceId = `serviceId:hash8` (first 8 chars of SHA256 of workdir)

This means starting the same service in different workdirs creates separate tracked instances.

## The ServiceManager in Commands

Commands access the service manager via `context.services`:

```typescript
// In a command's execute function
const status = context.services!.list('my-service');
const instanceId = await context.services!.start('my-service', { workdir: '/path' });
const logs = context.services!.logs('my-service', instanceId, 100);
```

Always null-check `context.services` — it's `undefined` when no services are configured in `defineCli`.

## Auto-Generated Management Commands

When you register a service in `defineCli({ services: [...] })`, the framework auto-generates these commands:

- `{id}-start` — Start with optional params, workdir, and instance name
- `{id}-stop` — Stop by instance ID (or the default instance)
- `{id}-restart` — Restart preserving the original workdir and params
- `{id}-ps` — Show status of all instances
- `{id}-logs` — Tail the log file (default 50 lines)
- `{id}-manage` — Interactive management panel (REPL only)
- `{id}-ls` — List discovered projects (only if `projectScanner` is defined)

You don't need to write these commands yourself — they are injected automatically.

## Common Pitfalls

- **Patterns match stdout AND stderr** — some tools (Next.js, Vite) log to stderr; your patterns will still match
- **Regex must match a single line** — patterns don't span lines; design them to match one log line
- **Capture groups are 1-indexed** — group 0 is the full match, group 1 is the first capture
- **No readiness config = immediately ready** — if you omit `readiness`, the service is marked ready right after spawn; use `readiness: { pattern: /.../, timeout: 30_000 }` to wait for actual readiness
- **Dynamic ports need dynamic preflight** — use `checkPort((ctx) => ctx.params?.port as number ?? 3000)` when ports come from params
- **Forgetting `checkCommand`** — the service fails cryptically if the binary isn't installed; add a preflight check with a helpful install hint
- **Timeout too short** — services that compile on first start (Next.js, Webpack) need longer timeouts (60-90s)
- **Static port in preflight but dynamic in params** — if the port comes from `paramsSchema`, use a function in `checkPort`, not a literal number

## When to Use paramsSchema

Add `paramsSchema` (a Zod schema) when the service needs runtime parameters that vary per invocation — ports, feature flags, URLs. These become flags on the auto-generated `{id}-start` command and are passed to both the `command()` and `env()` functions. Use `z.coerce.number()` for port parameters since CLI flags are always strings.

## When to Use projectScanner

Add `projectScanner` when one service definition manages multiple project directories (e.g., a dev server that can run in any project folder). The scanner enables the `{id}-ls` command for project discovery and integrates with `services.scanProjects()`. The scanner performs breadth-first search and prunes at foreign project boundaries.

## Typed Captures and MCP Discovery

Typed captures with `type: 'api-mcp'` make the captured endpoint discoverable by the agent integration. When the agent boots, it can scan service endpoints for MCP servers and automatically load their tools. This is the bridge between services and agent capabilities — you define the service, capture the MCP URL with the right type, and the agent picks it up.

## Registration in Codegen-Managed Projects

In codegen-managed projects, do NOT manually edit `cli.ts` marker regions to register services. Instead:

1. Write the service definition file in `src/services/`
2. Add the service to `project-spec.json`
3. Run `{{commands.clint-project-regen.id}}` to regenerate `cli.ts` (updates `@clint:begin services`, `@clint:begin imports`, and `@clint:begin events`)
4. Run `{{commands.clint-project-build.id}}` to verify

Import `defineService` from `../framework.js` (not from `@powerhousedao/ph-clint`) for automatic config typing.
