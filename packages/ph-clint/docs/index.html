Getting Started

Introduction

ph-clint (Powerhouse Command Line Intelligence) is a TypeScript framework for building CLI tools that double as AI agent harnesses. Define your commands once with Zod schemas, and ph-clint gives you a CLI, an interactive REPL, an MCP server, and agent tool integration — all from the same code.

One definition, four interfaces

A single defineCommand() call produces:

  • CLI subcommand — mycli greet --name World
  • REPL command — /greet --name World in interactive mode
  • MCP tool — exposed automatically when serving via MCP transport
  • Agent tool — available to Mastra AI agents as callable tools

Key capabilities

FeatureDescription
Zod-based schemasType-safe inputs with auto-generated help, flags, and validation
6-layer configurationEnv vars, config files, schema defaults — merged and typed
Interactive REPLInk-powered terminal with auto-completion and parameter prompting
Service managementStart, monitor, and restart long-running processes with readiness detection
Routine loopTick-based automation with pluggable triggers
AI agent integrationOptional Mastra agent with streaming, memory, workspace, and MCP tool discovery
TestabilityInjectable I/O at every level — no mocks needed

Tech stack

  • TypeScript 6.0, Node.js ≥ 22.13, ESM
  • Zod 4.3 for all schemas
  • Commander for CLI parsing, Ink + React for the REPL
  • Mastra (optional) for agents, memory, MCP

Quick Start

Installation

pnpm add @powerhousedao/ph-clint zod

Minimal CLI

#!/usr/bin/env node
import { defineCli, defineCommand } from '@powerhousedao/ph-clint';
import { z } from 'zod';

const greet = defineCommand({
  id: 'greet',
  description: 'Greet someone by name',
  inputSchema: z.object({
    name: z.string().describe('Name of the person to greet'),
    loud: z.boolean().default(false).describe('Shout the greeting'),
  }),
  execute: async ({ name, loud }) => {
    const msg = `Hello, ${name}!`;
    return loud ? msg.toUpperCase() : msg;
  },
});

const cli = defineCli({
  name: 'hello',
  version: '1.0.0',
  description: 'A minimal ph-clint example',
  commands: [greet],
  interactive: {
    welcome: 'Hello World CLI — type /help for commands',
  },
});

cli.run(process.argv);

This gives you:

  • Command mode: hello greet --name World --loud
  • Interactive mode: hello -i, then type /greet --name World
  • Built-in help: hello --help, hello greet --help, or /help in REPL
  • Auto-completion: Tab completion for commands, flags, and enum values

Core Concepts

Commands

Commands are the atomic unit. A command is defined once and works as:

  • A CLI subcommand (mycli greet --name World)
  • A REPL command (/greet --name World)
  • An MCP tool (when served via MCP transport)
  • An AI agent tool (when using Mastra integration)

Two Modes

ModeEntryBehavior
Commandmycli cmd --argsOne-shot execution, then exit
Interactivemycli -iPersistent REPL with /command syntax

Both modes share the same command definitions and routing logic.

Workspace vs Context

  • Workspace — the user's working directory (defaults to cwd, configurable via --workdir)
  • Context folder — {workspace}/.ph/ is ph-clint's managed state area (config, databases, persisted data)
Developer Guide

Defining Commands

Basic Command

import { defineCommand } from '@powerhousedao/ph-clint';
import { z } from 'zod';

export const greet = defineCommand({
  id: 'greet',
  description: 'Greet someone by name',
  inputSchema: z.object({
    name: z.string().describe('Name of the person to greet'),
    loud: z.boolean().default(false).describe('Shout the greeting'),
  }),
  execute: async ({ name, loud }) => {
    const msg = `Hello, ${name}!`;
    return loud ? msg.toUpperCase() : msg;
  },
});

The inputSchema is a Zod object schema. Each field becomes a CLI flag: --name, --loud. Field descriptions from .describe() are used in generated help text.

Return Values

Commands can return:

  • A string — displayed directly
  • An object { text, data } — text is displayed to the user, data is the structured result (useful for agent tool calls and programmatic access)
execute: async ({ title }, { workspace, config }) => {
  const task = { id: crypto.randomUUID(), title };
  await workspace.storeJsonObject('tasks.json', [task]);
  return { text: `Added: ${task.title}`, data: task };
},

CommandContext

The execute function receives input as its first argument and CommandContext as its second:

execute: async (input, context) => {
  context.workdir     // Resolved working directory (string)
  context.workspace   // WorkdirStore — persistent file-based state
  context.config      // Typed config object (from configSchema)
  context.stdout      // Output function for streaming text
  context.log         // Logger (debug/info/warn/error)
  context.routine     // Routine loop control (if triggers configured)
  context.processes   // ProcessManager (run shell commands)
  context.services    // ServiceManager (if services configured)
  context.emit        // Event emitter function
}

Parameter Prompting

In interactive mode, commands can prompt for missing parameters:

export const add = defineCommand({
  id: 'add',
  description: 'Add a new task',
  inputSchema: z.object({
    title: z.string().describe('Task title'),
    priority: z.enum(['low', 'medium', 'high']).optional()
      .describe('Task priority'),
  }),
  prompt: {
    promptForDefaults: false,      // Don't prompt for fields with defaults
    promptOptional: ['priority'],  // Prompt for these optional fields
  },
  execute: async ({ title, priority }, { workspace, config }) => {
    const p = priority ?? config.defaultPriority;
    // ...
  },
});

Typed Commands

Commands support generic type parameters for full type safety:

const inputSchema = z.object({
  name: z.string().regex(/^[a-zA-Z0-9-]+$/),
});

const init = defineCommand<typeof inputSchema, { text: string }, Config>({
  id: 'init',
  description: 'Initialize a project',
  inputSchema,
  execute: async ({ name }, { config }) => {
    // `name` is typed as string, `config` is typed as Config
    return { text: `Created ${name}` };
  },
});

Defining a CLI

Full Options

const cli = defineCli({
  name: 'mycli',
  version: '1.0.0',
  description: 'My CLI tool',               // Static string or Resolvable
  commands: [cmd1, cmd2],
  configSchema: myConfigSchema,               // Zod schema for typed config
  interactive: {
    welcome: 'My CLI — type /help',           // Static string or Resolvable
  },
  triggers: [myTrigger],                      // Routine loop triggers
  routine: { tickInterval: 2000 },            // Routine loop timing
  services: [myService],                      // Managed background services
  events: { 'service:ready': handler },       // Event handlers
  secretsSchema: mySecretsSchema,             // Sensitive config (censored in output)
  prompts: {                                   // Agent skill management
    artifacts: ['./skills', './dist/skills'],
    agents: { 'my-agent': { skills: ['skill-a'] } },
    skills: { 'skill-a': 'Skill description' },
  },
  workdir: '/custom/default/path',            // Implementation-level override
  configDefaults: { key: 'value' },           // Default config values
});

Resolvable Values

description and interactive.welcome accept either a static string or a function that receives the resolved config and workdir:

interactive: {
  welcome: ({ config, workdir }) => {
    const mode = config.apiKey ? 'LLM mode' : 'demo mode';
    return `My CLI (${mode})\nWorking in: ${workdir}`;
  },
},

The Cli Object

defineCli() returns a Cli object with these methods:

cli.run(argv, options?)          // Run CLI (command mode or REPL)
cli.execute(commandId, args)     // Execute a command directly
cli.parseArgs(commandId, argv)   // Parse CLI args for a command
cli.getCommand(id)               // Get a command by ID
cli.listCommands()               // List all registered commands
cli.generateHelp()               // Generate help text
cli.generateCommandHelp(id)      // Help for a specific command
cli.generateCompletion(shell)    // Shell completion script ('bash'|'zsh'|'fish')
cli.configEnvVars()              // List config env var mappings
cli.getMetadata()                // JSON-serializable CLI metadata
cli.configureAgent(loader)        // Set lazy agent loader (for Mastra)

RunOptions

cli.run() accepts optional RunOptions for testing and embedding:

cli.run(process.argv, {
  exit: (code) => { /* override process.exit */ },
  stdout: (msg) => { /* capture stdout */ },
  stderr: (msg) => { /* capture stderr */ },
  interactiveInput: asyncIterable,  // Headless REPL mode
  resume: 'thread-id',             // Resume agent conversation
  workdir: '/override/path',       // Override working directory
  configFile: './myconfig.json',   // Override config file path
  logLevel: 'debug',               // 'debug' | 'info' | 'warn' | 'error'
});

Configuration

Defining a Config Schema

const configSchema = z.object({
  defaultPriority: z.enum(['low', 'medium', 'high']).default('medium')
    .describe('Default priority for new tasks'),
  apiKey: z.string().optional()
    .describe('API key for the service'),
  maxRetries: z.number().default(3)
    .describe('Maximum retry attempts'),
});

6-Layer Resolution

Configuration is resolved by merging 6 layers, from highest to lowest priority:

PrioritySourceExample
1 (highest)--config flagmycli --config ./custom.json cmd
2Environment variablesTASKS_DEFAULT_PRIORITY=high
3Local config file{workdir}/.ph/tasks.config.local.json
4User config file~/.ph/tasks.config.user.json
5Implementation defaultsconfigDefaults in defineCli()
6 (lowest)Zod schema defaults.default('medium') in schema

Env vars are derived from the CLI name and field name in UPPER_SNAKE_CASE:

Type-Safe Config Access

import type { InferConfig } from '@powerhousedao/ph-clint';

const configSchema = z.object({
  connectPort: z.number().default(3000),
  apiKey: z.string().optional(),
});

type Config = InferConfig<typeof configSchema>;
// Config = { connectPort: number; apiKey?: string }

Secrets Schema

Use secretsSchema for sensitive config values (API keys, tokens). Values are merged into the config object but censored in help output, config command display, and metadata:

const cli = defineCli({
  name: 'assist',
  configSchema: z.object({
    model: z.string().default('anthropic/claude-haiku-4-5').describe('LLM model'),
  }),
  secretsSchema: z.object({
    apiKey: z.string().optional().describe('Anthropic API key'),
  }),
});

Secret fields are marked sensitive: true in CLI metadata and censored as *** in config command output.

Tip
When configSchema is present, a built-in config command is auto-registered for viewing and modifying settings: mycli config, mycli config --set defaultPriority=high, or /config in the REPL.

Workspace Persistence

The WorkdirStore provides file-based persistence rooted at {workdir}/.ph/{cliName}/:

execute: async (input, { workspace }) => {
  // Load JSON (returns fallback if file doesn't exist)
  const tasks = await workspace.loadJsonObject<Task[]>('tasks.json', []);

  // Modify and save
  tasks.push(newTask);
  await workspace.storeJsonObject('tasks.json', tasks);

  // Get paths
  workspace.getWorkdir();                  // User's working directory
  workspace.getStoreFolder();              // {workdir}/.ph/{cliName}/
  workspace.getStoreFolder('custom/dir');  // {workdir}/.ph/{cliName}/custom/dir
  workspace.getLocalConfigPath();          // {workdir}/.ph/{cliName}.config.local.json
}

For testing, use the in-memory store:

import { createMemoryWorkdirStore } from '@powerhousedao/ph-clint';

const workspace = createMemoryWorkdirStore();
// Same API, no filesystem I/O

Interactive Mode (REPL)

Enter with mycli -i. In the REPL:

  • /command --args — Run a command
  • /help — Show available commands
  • /exit or Ctrl+D — Exit
  • Bare text — Routed to the default subcommand (typically the agent, if configured)

Auto-Completion

The REPL provides automatic completion for:

  • Command names: /gr → Tab → /greet
  • Flag names: /add -- → Tab → --title, --priority, --due
  • Enum values: /list --filter d → Tab → done

Programmatic REPL (Testing)

import { createReplSession, createMemoryWorkdirStore } from '@powerhousedao/ph-clint';

const session = createReplSession({
  cli,
  context: {
    workdir: '/tmp/test',
    workspace: createMemoryWorkdirStore(),
    config: { defaultPriority: 'medium' },
    stdout: () => {},
  },
});

const result = await session.processInput('/greet --name World');
// result.text === 'Hello, World!'
// result.type === 'result'

Headless REPL (RunOptions)

async function* inputGenerator() {
  yield '/greet --name World';
  yield '/exit';
}

await cli.run(['node', 'cli.ts', '-i'], {
  interactiveInput: inputGenerator(),
  stdout: (msg) => console.log(msg),
});

Input Modes

The REPL supports three input modes:

In piped mode, the welcome message is printed to stdout (downstream consumers can filter it). The interactiveInput option in RunOptions is the testing equivalent of piped stdin.

Triggers & Routine Loop

Triggers are polling-based condition checks that produce work items for automatic execution.

Defining a Trigger

import { defineTrigger } from '@powerhousedao/ph-clint';

const fileChangeTrigger = defineTrigger({
  id: 'file-change',
  type: 'condition',
  setup: async (context) => {
    // Initialize trigger state (runs once when routine starts)
    context.state.lastModified = Date.now();
  },
  poll: async (context) => {
    // Called on every tick — return a WorkItem or null
    const current = await getLatestMtime(context.config.watchDir);

    if (current > context.state.lastModified) {
      context.state.lastModified = current;
      return {
        type: 'command',
        params: { commandId: 'build', args: {} },
        callbacks: {
          onSuccess: () => context.emit('build:complete'),
          onFailure: (err) => context.emit('build:failed', err),
        },
      };
    }
    return null;
  },
});

TriggerContext

PropertyDescription
context.contextCore context (config, emit, services, workspace, etc.)
context.statePersistent per-trigger state (survives across polls)
context.reactor()Lazy accessor — returns ReactorContext or undefined
context.agent()Lazy accessor — returns AgentProvider or undefined

Access config and emit via context.context.config and context.context.emit.

WorkItem

interface WorkItem {
  type: 'command' | 'function';
  params: Record<string, unknown>;  // For 'command': { commandId, args }
  callbacks?: {
    onSuccess?: (result: unknown) => void;
    onFailure?: (error: Error) => void;
  };
}

Registering Triggers

const cli = defineCli({
  name: 'watcher',
  commands: [build, watch, status],
  triggers: [fileChangeTrigger],
  routine: {
    tickInterval: 1000,   // Poll every 1s (default: 2000)
    idleInterval: 500,    // Wait between empty polls (default: 500)
  },
});

Controlling from Commands

const watch = defineCommand({

ProcessManager API

// Run a command, wait for completion
const result = await processes.run('npm run build', { label: 'build', timeout: 60_000 });
result.success   // boolean
result.output    // string (stdout + stderr)

const handles = processes.list();
handles.forEach(h => {
  h.label    // 'build'
  h.status   // 'running' | 'succeeded' | 'failed'

});

Service Management

Services are long-running background processes (dev servers, databases, etc.) with readiness detection, health checks, and lifecycle management.

Defining a Service

import { defineService, checkWorkdir, checkCommand, checkPort } from '@powerhousedao/ph-clint';

const devServer = defineService<Config>({
  id: 'dev-server',
  name: 'Dev Server',
  description: 'Local development server',

  // Static string or dynamic function
  command: 'npm run dev',
  // command: (params) => `npm run dev -- --port ${params?.port ?? 3000}`,

  // Environment variables from config
  env: (config) => ({
    PORT: String(config.port),
    NODE_ENV: 'development',
  }),

  // Optional: Zod schema for runtime parameters
  paramsSchema: z.object({
    port: z.coerce.number().optional(),
  }),

  readiness: { /* see Readiness Detection */ },
  preflight: [ /* see Preflight Checks */ ],

  shutdown: { signal: 'SIGTERM', timeout: 10_000 },
  restart: { enabled: true, maxRetries: 3, delay: 5_000 },
  maxInstances: 1,

  // Project scanner — auto-discovers projects in a directory tree
  projectScanner: {
    isProjectFolder: (dir) => fs.existsSync(path.join(dir, 'powerhouse.config.ts')),
    getProjectName: (dir) => path.basename(dir),
  },
});

Register with services: [devServer] in defineCli(). This auto-registers service commands: mycli dev-server --action up|down|restart.

Readiness Detection

Service is "ready" when all patterns match in stdout. Patterns can capture endpoints:

readiness: {
  patterns: [
    {
      name: 'http-port',
      pattern: /Local:\s*http:\/\/localhost:(\d+)/,
      captures: { 'http': 1 },              // Simple: group index
    },
    {
      name: 'mcp-server',
      pattern: /MCP server at (https?:\/\/[^\s]+)/,
      captures: {
        'mcp': { group: 1, type: 'api-mcp' },  // Typed capture
      },
    },
  ],
  timeout: 30_000,
}

Endpoint types: 'api-mcp', 'api-rest', 'api-graphql', 'website', 'other'

Captured endpoints are available via services.list():

const instances = services.list('dev-server');
instances[0].endpoints      // { 'http': 'http://localhost:3000' }
instances[0].endpointTypes  // { 'mcp': 'api-mcp' }

Preflight Checks

Three built-in check factories run before a service starts:

// Check working directory condition
checkWorkdir(
  (cwd) => fs.existsSync(path.join(cwd, 'package.json')),
  'Not a valid project directory',
  'Run from a directory with package.json',
)

// Check that a CLI tool is installed
checkCommand('docker', {
  hint: 'Install Docker: https://docs.docker.com/get-docker/',
  versionFlag: '--version',
  versionTest: (v) => parseInt(v) >= 20,
})

// Check that a port is free
checkPort(3000, 'Dev server')

AI Agent Integration (Mastra)

ph-clint supports optional AI agent integration via Mastra. The integration is fully lazy-loaded — Mastra is only imported when actually used.

Setting Up an Agent

const cli = defineCli({
  name: 'assist',
  configSchema: z.object({
    apiKey: z.string().optional().describe('Anthropic API key'),
    model: z.string().default('anthropic/claude-haiku-4-5')
      .describe('LLM model'),
  }),
  commands: [cmd1, cmd2],
  interactive: {
    welcome: ({ config }) => {
      const mode = config.apiKey ? `LLM: ${config.model}` : 'demo mode';
      return `Assistant (${mode})`;
    },
  },
});

cli.configureAgent(async (ctx) => {
  // Return demo agent when no API key
  if (!ctx.config.apiKey) return createDemoAgent();

  // Lazy-load Mastra only when needed
  const { createMastraHelpers } = await import('@powerhousedao/ph-clint/mastra');
  const { Agent } = await import('@mastra/core/agent');
  const m = createMastraHelpers(ctx);

  return m.wrapAgent(new Agent({
    id: 'assistant',
    name: 'My Assistant',
    instructions: 'You are a helpful assistant.',
    model: ctx.config.model,
    tools: await m.getTools(),
    workspace: await m.createWorkspace(),
    memory: await m.createMemory(),
  }));
});

AgentSetupContext

interface AgentSetupContext<TConfig> {
  workdir: string;         // Working directory
  config: TConfig;         // Resolved config
  cliName: string;         // CLI name
  cliVersion: string;      // CLI version
  context: CommandContext;  // Full command context
  commands: Command[];     // All registered commands
  skills: SkillInfo[];     // Skills assigned to this agent
}

MastraHelpers

createMastraHelpers(ctx) returns:

MethodDescription
getTools(options?)Converts CLI commands to Mastra tools. With includeMcp: true (default), also discovers MCP tools from running services.
getAgentInstructions(agentId)Loads compiled agent profile instructions for the given agent ID from prompts config.
createWorkspace()Creates a Mastra Workspace with LocalFilesystem rooted at workdir.
createMemory()Creates thread-based Memory backed by LibSQL at {workspace}/.ph/{cliName}/.mastra/db/mastra.db.
wrapAgent(agent, opts?)Wraps a Mastra Agent as an AgentProvider yielding StreamChunks. maxSteps defaults to 30.

MCP Tool Discovery

When services with api-mcp endpoints are running, getTools() discovers their MCP tools:

import { MCPClient } from '@mastra/mcp';

const tools = await m.getTools({ MCPClient });
// Contains CLI command tools + MCP-discovered tools
// MCP tools prefixed: '{serviceId}-mcp__toolName'

Demo Agent

Implement AgentProvider directly for testing or when no API key is available:

import type { AgentProvider, StreamChunk } from '@powerhousedao/ph-clint';

function createDemoAgent(): AgentProvider {
  return {
    id: 'demo',
    async *stream(prompt, opts) {
      yield {
        type: 'text-delta',
        text: `Echo: ${prompt}`,
      } satisfies StreamChunk;
    },
  };
}
Agent in the REPL
When an agent is configured, bare text (not prefixed with /) is routed to the agent. Use --resume thread_id to continue a previous conversation.

Events

ph-clint uses a central event bus for decoupled communication:

const cli = defineCli({
  events: {
    'service:ready': (data) => { /* ... */ },
    'build:complete': () => { /* ... */ },
    'custom:event': (data) => { /* ... */ },
  },
});

Events can be emitted from commands (context.emit('my-event', data)), triggers (context.emit('build:complete')), and services (automatic lifecycle events).

Every CLI exposes machine-readable metadata via getMetadata() and the --meta global flag:

mycli --meta          // Outputs JSON metadata to stdout

Skill Templates

Skills are reusable agent instruction packages. Each skill is a folder containing a SKILL.md Handlebars template (with YAML frontmatter) and an optional references/ directory with supporting documents. Skills are configured in the prompts section of defineCli(), auto-installed into the workspace on first run, and exposed as CLI commands (/skill-name --prompt "...") that route to the agent.

Configuration

Skills are configured via PromptsConfig on defineCli():

import { defineCli } from '@powerhousedao/ph-clint';

defineCli({
  // ...other config...
  prompts: {
    // Directories containing built skill folders (first existing wins)
    artifacts: [
      path.join(PROJECT_ROOT, 'gen', 'skills'),
      path.join(PROJECT_ROOT, 'dist', 'gen', 'skills'),
    ],

    // Agent profiles: instruction sections + skill assignments
    agents: {
      'rupert-dev-agent': {
        name: 'RupertDevAgent',
        sections: ['AgentBase.md', 'ReactorPackageDevAgent.md'],
        skills: ['document-modeling', 'document-editor-creation'],
      },
    },

    // Skill definitions: description + optional inputSchema + instructionTemplate
    skills: {
      'document-modeling': {
        description: 'Design Powerhouse document models with state schemas and reducers',
        inputSchema: z.object({
          mode: z.enum(['expert', 'discovery', 'one-shot']).default('expert'),
        }),
        instructionTemplate: 'Use your {{skillId}} skill in {{mode}} mode for: {{prompt}}',
      },
      // Plain string shorthand: equivalent to { description: '...' }
      'playwright-cli': 'Automate browser interactions and Playwright tests',
    },
  },
});
Tip
The inputSchema fields become additional command parameters. When a user runs /document-modeling --prompt "...", the mode parameter is prompted or accepted as a flag. The instructionTemplate is a Handlebars template that receives { skillId, description, prompt, ...inputValues }.

Folder Structure

Skill source templates live in a prompts/ directory. There are two kinds of skill sources:

  • Template skills (skills-tpl/) — multi-section Handlebars templates compiled at build time
  • External skills (skills-ext/) — pre-written SKILL.md files with optional reference documents
prompts/
  agent-profiles/              # Agent instruction section templates
    AgentBase.md
    ReactorPackageDevAgent.md
  skills-tpl/                  # Template skills (compiled at build)
    document-modeling/
      .cli-docs.md             # CLI documentation context (auto-injected)
      .preamble.md             # Optional preamble section
      .result.md               # Optional result format section
      00.check-prerequisites.md
      01.write-description.md
      03.implement-reducers.md
    document-editor-creation/
      .cli-docs.md
      00.check-prerequisites.md
      01.write-editor-implementation.md
  skills-ext/                  # External skills (copied as-is)
    playwright-cli/
      SKILL.md                 # Handlebars template with frontmatter
      references/              # Reference docs included in template
        session-management.md
        test-generation.md
  skills-lock.json             # Tracks installed skill versions

Each SKILL.md has YAML frontmatter with metadata:

---
name: document-modeling
description: "Design Powerhouse document models with state schemas and reducers"
metadata:
  author: Powerhouse
  version: "1.0.0"
compatibility: "Node.js >=22.13, pnpm"
---

Build Process

Skill templates are compiled at build time using buildSkills() from ph-clint-dev:

import { buildSkills } from 'ph-clint-dev';

buildSkills({
  include: [path.join(PROJECT_ROOT, 'prompts')],
  context: {                            // Template variables
    workspaceDir,
    connectPort,
    switchboardPort,
  },
  output: [                             // Output directories
    path.join(PROJECT_ROOT, 'gen'),
    path.join(PROJECT_ROOT, 'dist', 'gen'),
  ],
  cli,                                  // CLI instance for metadata injection
});

The build process:

  1. Reads skills-tpl/ directories — each numbered section file (00.*.md, 01.*.md, ...) is compiled individually into references/, and a SKILL.md is generated with frontmatter + optional preamble + links to those references
  2. Copies skills-ext/ skills as-is (SKILL.md + references/)
  3. Compiles all Handlebars templates with the provided context variables
  4. Compiles agent profile sections from agent-profiles/ into instruction strings
  5. Copies .cli-docs.md files (rendered through Handlebars) alongside skills
  6. Writes compiled output to gen/ and dist/gen/

Add a build script to package.json:

"build:skills": "tsx scripts/build-skills.ts"

The prompts.artifacts config should point to the output directories so the CLI finds the compiled skills at runtime.

Typed Factories (createTypes)

When building a CLI with a config schema and/or a Powerhouse document registry, createTypes() returns pre-typed versions of defineCommand, defineTrigger, defineService, and createDocumentChangeTrigger — no need to repeat generic type parameters at every call site:

import { createTypes, defineRegistry } from '@powerhousedao/ph-clint';
import { z } from 'zod';
import { MyDocModel } from './document-models/my-doc.js';

const configSchema = z.object({
  switchboardPort: z.number().default(4001),
});

const registry = defineRegistry([MyDocModel] as const);

const { defineCommand, defineTrigger, defineService, createDocumentChangeTrigger } =
  createTypes({ configSchema, registry });

// ctx.config is typed as { switchboardPort: number }
// ctx.reactor().client is typed with registry-derived methods
const myCmd = defineCommand({
  id: 'my-cmd',
  description: 'A typed command',
  inputSchema: z.object({ name: z.string() }),
  execute: async ({ name }, ctx) => {
    ctx.config.switchboardPort;  // number (typed)
    return `Hello ${name}`;
  },
});

Powerhouse Integration

ph-clint integrates with the Powerhouse document reactor via @powerhousedao/reactor. The integration is optional — Powerhouse dependencies are only loaded when features are configured.

Document Registry

Define a typed registry from your document model modules:

import { defineRegistry } from '@powerhousedao/ph-clint';
import { MyDocModel } from './document-models/my-doc.js';
import { OtherDocModel } from './document-models/other-doc.js';

const registry = defineRegistry([MyDocModel, OtherDocModel] as const);
// Type: { 'my-doc-type': RegistryEntry<MyState, MyActions>, ... }

The registry enables type-safe document operations — client.get(), client.execute(), etc. are narrowed by document type.

Building the Reactor

Use buildDefaultReactor() in the CLI’s configureReactor.create callback:

const cli = defineCli({
  name: 'my-agent',

    documentModels: [MyDocModel],

      documentModels: [MyDocModel],
      drive: { name: 'my-drive' },
      subscriptions: {
        documentTypes: ['my-doc-type'],

      },
    }),

});

Multi-Drive Support

Configure multiple drives with personal and watched roles:

buildDefaultReactor(ctx, {
  documentModels: [MyDocModel],
  drives: [
    { name: 'my-projects', role: 'personal' },
    { name: 'shared-specs', role: 'watched',
      remoteUrl: 'https://switchboard.example.com/d/shared' },
  ],
});
  • personal — the agent’s own drive for creating/editing documents. The first personal drive sets driveId and personalDriveId on ReactorContext.
  • watched — remote drives synced for reading. Created via ensureRemoteDrive().

Folder Operations

The FolderOperations API provides a file-system-like interface over a drive’s folder hierarchy:

import { createFolderOperations } from '@powerhousedao/ph-clint';

const folders = createFolderOperations(reactor.client, reactor.personalDriveId);

// List contents of a folder
const entries = await folders.listFolder('projects/active');

// Create a nested folder
await folders.createFolder('projects/active/new-project');

// Find a document by name
const doc = await folders.findDocument('projects/active', 'my-spec');

Auto-generated folder commands can be added with createFolderCommands().

Document Change Trigger

Watch for document changes and react automatically:

const { createDocumentChangeTrigger } = createTypes({ configSchema, registry });

const specChangeTrigger = createDocumentChangeTrigger({
  id: 'spec-change',
  documentType: 'my-doc-type',  // or ['type-a', 'type-b']
  initialReconcile: true,
  onChange: async (docs, ctx) => {
    return {
      type: 'command',
      params: { commandId: 'regen', args: {} },
    };
  },
});

Options: documentType (single or array), documentId (string or async resolver), initialReconcile (default: true), filter (per-document predicate).

Bootstrapping

Cli.bootstrap() resolves workdir, config, and context without entering the full run() pipeline. Useful for codegen scripts or programmatic access:

const cli = defineCli({ name: 'my-cli', configSchema, commands: [...] });

const { workdir, config, context } = await cli.bootstrap({
  workdir: '/path/to/project',
  logLevel: 'warn',
});

Accepts BootstrapOptions: workdir, stdout, stderr, logLevel.

export const add = defineCommand({
  id: 'add',
  description: 'Add a new task',
  inputSchema: z.object({
    title: z.string().describe('Task title'),
    priority: z.enum(['low', 'medium', 'high']).optional()
      .describe('Task priority'),
    due: z.string().optional().describe('Due date (YYYY-MM-DD)'),
  }),
  prompt: {
    promptForDefaults: false,
    promptOptional: ['priority'],
  },
  execute: async ({ title, priority, due }, { workspace, config }) => {
    const tasks = await workspace.loadJsonObject<Task[]>('tasks.json', []);
    const task: Task = {
      id: crypto.randomUUID(),
      title,
      priority: priority ?? (config.defaultPriority as Task['priority']),
      due: due ?? null,
      done: false,
    };
    tasks.push(task);
    await workspace.storeJsonObject('tasks.json', tasks);
    return { text: `Added: ${task.title} [${task.priority}]`, data: task };
  },
});

03 — File Watcher

File change detection with automatic build triggering via the routine loop.

src/trigger.ts

import { defineTrigger } from '@powerhousedao/ph-clint';

export const fileChangeTrigger = defineTrigger({
  id: 'file-change',
  type: 'condition',
  setup: async (context) => {
    context.state.lastModified = Date.now();
  },
  poll: async (context) => {
    const current = await getLatestMtime(context.config.watchDir);
    if (current > context.state.lastModified) {
      context.state.lastModified = current;
      return {
        type: 'command',
        params: { commandId: 'build', args: {} },
        callbacks: {
          onSuccess: () => context.emit('build:complete'),
          onFailure: (err) => context.emit('build:failed', err),
        },
      };
    }
    return null;
  },
});

src/commands/build.ts

export const build = defineCommand({
  id: 'build',
  description: 'Run the build command',
  inputSchema: z.object({}),
  execute: async (_, { config, processes }) => {
    const result = await processes!.run(config.buildCommand, {
      label: 'build', timeout: 60_000,
    });
    return { text: result.success ? 'Build succeeded' : 'Build failed' };
  },
});
